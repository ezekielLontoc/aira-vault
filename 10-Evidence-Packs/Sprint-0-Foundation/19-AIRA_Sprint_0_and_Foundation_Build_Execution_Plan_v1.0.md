# 1. Executive Summary

AIRA Sprint-0 is the controlled execution bridge between the approved architecture standards and the first production-bound development work. It is designed to establish a clean, governed, reproducible, and evidence-producing foundation before the team begins business-feature development.

The sprint focuses on foundation readiness: repository bootstrap, Golden Source controls, devcontainer and workstation standards, CI/CD gates, architecture fitness checks, security baseline, MicroFunction runtime skeleton, sample transaction, API and database contract patterns, observability hooks, and AI-assisted development controls.

| Executive Decision<br>Controlled foundation development may start under Sprint-0 once the Ready-to-Start Gate is approved. Business module coding must wait until the Ready-to-Code Gate and relevant module readiness criteria are satisfied. |

| --- |

## 1.1 Strategic Outcomes

| Outcome | Sprint-0 Delivery Mechanism |

| --- | --- |

| Controlled start | The team starts with approved repositories, branch protection, CODEOWNERS, templates, and CI gates. |

| Developer readiness | Every developer has a clean environment, approved toolchain, devcontainer, AI usage acknowledgement, and local validation path. |

| Architecture enforcement | SOLID, Clean Architecture, DDD, ports-and-adapters, dependency direction, and no-bypass rules are checked by code review and automated fitness functions. |

| Security from day one | Keycloak/AD, Vault/secrets pattern, SBAC/OPA hooks, no-secrets scanning, and classification-aware data handling are established early. |

| Testing discipline | TDD, deterministic unit tests, contract tests, ArchUnit checks, and evidence artifacts are required from the first mergeable code. |

| MicroFunction proof | A sample configurable transaction demonstrates runtime assembly, idempotency, audit, error handling, compensation, outbox, and safe response. |

| AI governance | ChatGPT/Codex/Claude usage follows CLAUDE.md hierarchy, Harness boundary, LiteLLM-only model routing for application flows, and human accountability. |

| Evidence readiness | Each sprint output carries AVCI evidence: owner, source, validation, classification, approval, and improvement path. |

## 1.2 What Sprint-0 Is Not

It is not a business-feature sprint. Business-specific modules should not proceed until contracts, acceptance criteria, classifications, and test cases are ready.

It is not a shortcut around architecture. Sprint-0 must prove the architecture controls before allowing higher delivery velocity.

It is not an autonomy license for AI tools. AI may draft and assist, but humans remain accountable and all controlled actions follow policy gates.

It is not a one-time setup task. Sprint-0 establishes repeatable patterns that must remain enforced throughout the project lifecycle.

# 2. Purpose, Scope, and Authority

## 2.1 Purpose

This plan defines the work, sequencing, acceptance criteria, roles, and evidence required to complete Sprint-0 for AIRA. It provides the actionable execution model that the Software Development, DevSecOps, Security, Infrastructure, DBA, QA, and AI Engineering teams can follow before regular feature delivery begins.

## 2.2 In Scope

Repository bootstrap, branch protection, CODEOWNERS, PR/MR templates, issue templates, labels, and baseline governance files.

Golden Source baseline for approved dependencies, container images, plugins, templates, prompts, guardrails, and pipeline artifacts.

Developer workstation and devcontainer readiness aligned with the Greenfield Environment Standard.

Initial CI/CD pipeline with test, scan, policy, architecture, and evidence gates.

Base service skeletons for gateway, workflow, retrieval, orchestrator, harness, admin-governance, and sample bounded-context service.

MicroFunction runtime skeleton and one sample transaction demonstrating standard controls and one isolated business function.

Initial contract registry, database migration baseline, observability baseline, security baseline, and AI governance baseline.

Sprint-0 evidence pack and closeout report.

## 2.3 Out of Scope

Full business module delivery, except a controlled sample transaction used to prove the framework.

Production deployment, customer data migration, external system go-live, or production AI model use.

Full DR/BCP, UAT, and production operations manuals, except starter runbook stubs needed for foundation readiness.

Unapproved direct integration to model providers, production databases, external customer systems, or unmanaged public repositories.

# 3. Sprint-0 Operating Principles

Sprint-0 must demonstrate that the AIRA delivery engine is governed before it becomes fast. The following rules are non-negotiable for all foundation work.

| ID | Principle | Sprint-0 Meaning |

| --- | --- | --- |

| S0-1 | AVCI by construction | Every Sprint-0 artifact must have owner, source, version, classification, verification evidence, and improvement path. |

| S0-2 | Architecture first | Repositories, services, packages, and pipelines must follow the approved architecture before business code is added. |

| S0-3 | SOLID and boundary enforcement | Base code must prove single responsibility, dependency direction, clean package boundaries, ports/adapters separation, and no infrastructure leakage into domain logic. |

| S0-4 | Configuration-first | Runtime behavior should be expressed through approved configuration, templates, policies, and MicroFunction catalog entries wherever feasible. |

| S0-5 | Secure by default | No secret, credential, PII, direct provider key, or unrestricted administrative path may enter the repository or logs. |

| S0-6 | TDD and deterministic evidence | Mergeable code must include deterministic unit tests, contract tests where relevant, and CI evidence. |

| S0-7 | Fail-safe, not fail-open | If identity, policy, guardrails, secrets, or audit dependencies are unavailable, controlled operations are blocked. |

| S0-8 | One contribution path | All changes enter through protected Git workflow, review, CI gates, signed artifacts, and evidence retention. |

| S0-9 | AI assistance is bounded | AI drafts and analyzes; humans approve. Tool execution follows CLAUDE.md, Harness, SBAC, OPA, and audit boundaries. |

| S0-10 | Rebuildability and reversibility | Every foundation component must be reproducible from source and reversible through rollback, forward-fix, or rebuild procedure. |

## 10.3 Sample Transaction Scope

Sprint-0 should use one low-risk sample transaction to prove the framework. The transaction must be synthetic and must not use production customer data. A recommended example is SAMPLE-CASE-CREATE or SAMPLE-DOCUMENT-INTAKE-STUB.

| Step | Required Proof |

| --- | --- |

| Receive and correlate | Request receives trace_id, request_id, tenant placeholder, and correlation context. |

| Session and security | Actor context is resolved and authorization/policy is evaluated. |

| Parse, normalize, validate | Payload is transformed into typed request and invalid input is safely rejected. |

| Idempotency and concurrency | Duplicate request does not double-create business effect. |

| Business function | Only the STP-BUS-* sample function contains domain-specific behavior. |

| Persistence and outbox | State and event intent are committed safely through approved pattern. |

| Audit and observability | Audit record, log, trace, and metric share correlation IDs. |

| Error path | Failure is classified, retried or not retried by policy, compensated if required, and returned as safe error response. |

# 11. Risks, Dependencies, Assumptions, and Decisions

| Risk ID | Risk | Impact | Mitigation / Control |

| --- | --- | --- | --- |

| R1 | Team starts business coding before foundation gates pass. | Architecture drift, inconsistent patterns, rework. | Block business-feature stories until Ready-to-Code Gate is approved. |

| R2 | Repository controls are configured manually and not reproducibly documented. | Audit weakness and inconsistent future repositories. | Export settings, document branch protection, and store templates in Golden Source. |

| R3 | Developers bypass devcontainer or install unapproved tools locally. | Drift, supply-chain exposure, non-reproducible builds. | Greenfield controls, workstation validation, pre-commit and CI enforcement. |

| R4 | CI becomes too heavy and slows early iteration. | Team bypasses quality checks. | Use staged gates: fast local checks, required PR checks, heavier scheduled scans. |

| R5 | AI-generated code weakens architecture boundaries. | Maintainability and security risk. | CLAUDE.md rules, architecture tests, reviewer checklist, AI involvement declaration. |

| R6 | Security baseline is deferred. | Secrets, identity, and policy issues discovered late. | Require secret scan, OPA/SBAC scaffold, Vault pattern, and safe local profile in Sprint-0. |

| R7 | MicroFunction framework becomes over-engineered before proof. | Delay and complexity. | Build only the minimum vertical slice, then evolve through tests and ADRs. |

# 12. Sprint-0 Closeout and Handoff to Sprint 1

Sprint-0 closes with a formal readiness review. The review must demonstrate working controls and evidence, not only reported progress. The review should be conducted as a formal defense by the team: what was built, what was learned, what was improved, what risks remain, and whether Sprint 1 may begin.

| Closeout Item | Required Content |

| --- | --- |

| Foundation demo | Clone, build, run tests, run sample service, execute sample transaction, show CI evidence. |

| Evidence pack review | AVCI evidence, test reports, scan reports, architecture fitness, security review, decisions, waivers. |

| Open risk review | Unresolved issues, ownership, severity, target date, and whether they block Sprint 1. |

| Sprint 1 readiness | Approved backlog, module readiness criteria, owner assignments, and dependencies. |

| Lessons learned | What should be promoted to Obsidian/LLM Wiki after review and what should be improved in standards or templates. |

## 12.1 Ready-to-Code Decision Options

| Decision | Meaning | Allowed Next Step |

| --- | --- | --- |

| Approved | All critical gates pass; only non-blocking actions remain. | Start Sprint 1 controlled foundation and approved module work. |

| Approved with Conditions | Minor issues remain but have owners, expiry, and compensating controls. | Start limited Sprint 1 work that does not depend on open blockers. |

| Deferred | One or more critical gates failed or evidence is insufficient. | Continue Sprint-0 remediation; business coding remains blocked. |

| Rejected | Foundation approach violates architecture, security, or AVCI principles. | Escalate to Architecture Board and revise approach. |
