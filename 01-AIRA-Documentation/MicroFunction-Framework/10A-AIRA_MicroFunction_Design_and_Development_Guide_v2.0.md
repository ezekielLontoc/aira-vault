# 1. Executive Summary

The AIRA MicroFunction framework is a configuration-driven design pattern for building modular, maintainable, decoupled, runtime-assembled business processes. It replaces hardwired transaction code with reusable steps that are selected, ordered, parameterized, validated, cached, and executed by the framework.

Version 2.0 makes the guide explicit about enterprise-grade design discipline. A configured process is treated like code. A coded business MicroFunction is treated like an architecture artifact. Both must satisfy AVCI and the Enterprise Design Principles and SOLID Enforcement Layer.

| Design Philosophy<br>Configure the process. Reuse the standard functions. Code only the business logic gap. Govern every change through AVCI evidence. Preserve or improve SOLID compliance, Clean Architecture boundaries, DDD ownership, security, observability, testability, reversibility, and rollbackability. |

| --- |

| Outcome | Required Result |

| Governed extensibility | New code enters through catalog registration, tests, architecture review, and AVCI evidence. |

# 3. Design Intent and Development Boundary

The framework separates repeatable mechanics, configurable process behavior, business logic, and governance. This prevents every developer from implementing a different version of validation, audit, retry, model routing, or error handling.

| Area | Owner / Path | Meaning |

| --- | --- | --- |

| Framework Responsibility | Standard AIRA framework code | Receive, correlate, session, security, parse, normalize, validate, idempotency, locking, audit, logging, persistence envelope, eventing, retry, compensation, safe response, observability, guardrails, Harness execution. |

| Configuration Responsibility | Solutions Architect / authorized configuration owner | Which steps run, in what order, with which parameters, timeouts, retry policy, error policy, compensation, classification ceiling, and human approval route. |

| Business Responsibility | Software Developer and Business SME | Business rules, domain calculations, domain decisions, state transitions, and validations not yet expressible through rule tables or DMN. |

| No-Code Change | Preferred path | Use existing standard functions, parameters, rules, DMN, OPA policies, feature flags, and table bindings. |

| Low-Code Change | Controlled path | Add configuration rows, rule-table entries, DMN definitions, prompt templates, or guardrail policies through MR review. |

| Code-Required Change | Exception path | Create a new isolated STP-BUS-* or STP-STD-* MicroFunction only when the required capability does not yet exist. |

## 3.1 Developer Rules: What to Code and What Not to Code

| Rule | Guidance |

| --- | --- |

| Do Code | Business-specific MicroFunctions with a narrow contract; reusable new standard functions; repository methods behind approved data-access patterns; typed DTOs; deterministic tests; adapter implementations behind ports. |

| Do Not Code | Hardwired transaction sequences; direct Redis access inside business logic; direct Kafka publish inside business logic; direct HTTP response construction; direct audit writes; direct model provider calls; hidden retry loops; swallowed exceptions. |

| Always Use | MicroFunction interface, execution envelope, catalog registration, typed parameters, framework-managed audit, framework-managed error policy, configuration-driven assembly, and approved ports/adapters. |

| Escalate When | A new step category is needed; a destructive update is proposed; a function touches Restricted data; a model/tool action is proposed; a sequence bypasses mandatory standard controls. |

# 4. Enterprise Design Principles and SOLID Enforcement Layer

All MicroFunction design and development must comply with 01A. The principles are not advisory. They are design gates, implementation gates, test gates, and activation gates.

| No. | Principle | MicroFunction Design Requirement |

| --- | --- | --- |

| 1 | SOLID | Each function is small, interface-driven, substitutable, dependency-inverted, and extendable without modifying unrelated functions. |

| 2 | Clean Architecture | Domain logic remains independent from controllers, databases, model providers, workflow engines, and infrastructure adapters. |

| 3 | DDD / Bounded Contexts | A business function belongs to one bounded context and does not write into another context's schema or state. |

| 4 | Ports and Adapters | External systems, DB, cache, workflow, DMS, and model calls are behind approved ports and adapters. |

| 5 | DRY, KISS, YAGNI | Reuse standard steps; keep business functions simple; do not build speculative generic frameworks. |

| 6 | Idempotency by Design | Mutating functions declare idempotency key, replay behavior, and deduplication rules. |

| 7 | Determinism and Reproducibility | The same inputs, configuration version, and dependencies produce explainable outcomes; non-determinism is controlled. |

| 8 | Fail-Safe, Not Fail-Open | Missing policy, guardrail, audit, identity, or config blocks protected processing. |

| 9 | Human-in-the-Loop | High-risk, low-confidence, policy-exception, and destructive flows route to Flowable human review. |

| 10 | Least Privilege / SBAC | Functions and agents receive only the skills, scopes, and data access required. |

| 11 | Separation of Duties | Maker, checker, approver, promoter, and auditor responsibilities remain distinct. |

| 12 | Observability by Design | Every function emits trace, metric, log, and audit signals with correlation IDs and redaction. |

| 13 | Policy as Code | Business, security, routing, and admission decisions use OPA/DMN/rules where appropriate. |

| 14 | Testability by Design | Functions are testable in isolation with synthetic data, deterministic inputs, and mocked ports. |

| 15 | Secure by Design | Classification, validation, authorization, redaction, and secret handling are mandatory controls. |

| 16 | Resilience Patterns | External dependencies use timeout, retry, circuit breaker, fallback, DLQ, and compensation policies. |

| 17 | Architectural Fitness Functions | CI verifies boundaries, imports, contracts, idempotency, determinism, and policy compliance. |

| 18 | Progressive Autonomy | Automation starts advisory and earns more delegated scope only through evidence and trust. |

| 19 | Reversibility / Rollbackability | Changes include rollback, forward-fix, cache invalidation, compensation, or deactivation plan. |

| 20 | AVCI | Every design, configuration, function, test, and activation has owner, evidence, classification, and improvement path. |

## 4.1 SOLID Interpretation for Developers

| SOLID Principle | Correct MicroFunction Design | Rejected Design |

| --- | --- | --- |

| Single Responsibility | One business decision, validation, calculation, or state transition per function. | One function parses, validates, authorizes, writes DB, publishes Kafka, and builds response. |

| Open/Closed | New behavior is added through new registered functions, rules, parameters, or adapters. | Modifying a common standard function for one special business case. |

| Liskov Substitution | Any registered implementation honors the same input, output, error, idempotency, and evidence contract. | A function changes null/error semantics or skips audit compared with the contract. |

| Interface Segregation | Functions receive only the context slices and ports they need. | Functions depend on a giant global context or utility service. |

| Dependency Inversion | Business logic depends on domain ports and framework contracts. | Business logic imports database clients, Kafka producers, provider SDKs, or REST clients directly. |

# 5. Configuration-First Design Procedure

Solutions Architects must attempt design in the following order. Code is a controlled exception, not the default.

| Step | Action | Required Evidence |

| --- | --- | --- |

| 1 | Define intent: transaction purpose, actor, trigger, bounded context, data classification, and outcome. | Ticket, requirement, classification, owner, expected response, evidence target. |

| 2 | Select the closest assembly template: REST_SYNC, KAFKA_ASYNC, TEMPORAL_MACHINE, FLOWABLE_HUMAN, HYBRID, AI_RAG_QUERY, DOCUMENT_INTAKE, RECONCILIATION, RETENTION. | Template reference, decision note, sequence diagram link. |

| 3 | Reuse standard functions for all common concerns. | Catalog references from 10D, mandatory step check. |

| 4 | Model business rules as parameters, DMN, OPA policy, or rule tables where possible. | Rule version, test cases, approval owner. |

| 5 | Identify the exact business logic gap that still requires code. | Code-required justification, alternatives considered. |

| 6 | Design new STP-BUS-* or STP-STD-* function with SOLID, Clean Architecture, DDD, and testability gates. | Function contract, owner, bounded context, ports/adapters, tests. |

| 7 | Define error, retry, timeout, compensation, DLQ, human escalation, and safe response behavior. | Error policy, retry policy, compensation step, test evidence. |

| 8 | Publish configuration through MR, validation report, and activation approval. | Config diff, validator output, CI evidence, AVCI summary. |

## 5.1 No-Code / Low-Code / Code-Required Decision Matrix

| Scenario | Preferred Method | Approval / Evidence |

| --- | --- | --- |

| New transaction using existing steps | No-code configuration: txn_definition, step bindings, parameters, table bindings, retry and error policies. | Configuration MR, publish validation, AVCI summary, regression tests. |

| Change order, threshold, timeout, retry, approval route | No-code configuration change. | Impact analysis, test evidence, owner approval. |

| New business rule expressible as decision table | Low-code Flowable DMN or rule-table update. | Rule owner approval, sample inputs/outputs, negative tests. |

| New policy constraint | OPA policy-as-code change. | Security review, policy tests, audit mapping. |

| New domain operation not available in catalog | Code-required STP-BUS-* MicroFunction. | ADR/TDL if material, unit/contract tests, catalog registration. |

| New framework concern such as error category or cache strategy | Code-required standard STP-* framework function. | Architecture approval, regression tests, compatibility review. |

| New AI behavior | Prompt, guardrail, model-alias, or tool-action configuration first; code only if new action adapter is required. | Golden dataset evaluation, guardrail tests, Harness/SBAC validation. |

# 6. Standard Assembly Pattern

Every transaction is assembled from a standard sequence template. Steps may be mandatory, conditional, or optional. The publish validator must reject any activated configuration that omits mandatory framework controls or lacks a defined error path.

| # | Step Group | Step Prefix | Purpose |

| --- | --- | --- | --- |

| 01 | Receive | STP-RCV-* | Accept REST, Kafka, webhook, batch, file, or scheduled trigger. |

| 02 | Correlate | STP-COR-* | Create or propagate trace_id, request_id, correlation_id, causation_id. |

| 03 | Session / Actor | STP-SES-* | Resolve actor, tenant, role, branch/unit, service identity, or AI agent identity. |

| 04 | Rate / Quota | STP-RATE-* | Apply API rate limit, tenant quota, AI budget, or service throttle. |

| 05 | Security | STP-SEC-* | Authenticate, authorize, classify, evaluate RBAC/ABAC/SBAC/OPA. |

| 06 | Parse | STP-PRS-* | Convert JSON, XML, CSV, Avro, CloudEvents, or multipart input. |

| 07 | Normalize | STP-NRM-* | Canonicalize, sanitize, trim, encode/decode, detect unsafe content. |

| 08 | Validate | STP-VAL-* | Validate schema, fields, rules, DMN, and policy constraints. |

| 09 | Idempotency | STP-IDP-* | Check, reserve, replay, complete, or expire idempotency keys. |

| 10 | Concurrency | STP-CON-* | Apply optimistic lock, pessimistic lock, Redis lock, or version check. |

| 11 | Configuration | STP-CFG-* | Resolve active runtime parameters, feature flags, and process version. |

| 12 | Cache | STP-CAC-* | Read-through, write cache, invalidate cache, or bypass cache safely. |

| 13 | Business Logic | STP-BUS-* / STP-RUL-* | Execute business-specific function or configurable rule/DMN. |

| 14 | Persistence | STP-DB-* | Select, insert, update, upsert, append state, append history, or outbox. |

| 15 | Audit / History | STP-AUD-* / STP-HIS-* | Write immutable technical evidence and business-visible timeline. |

| 16 | Event / Notification | STP-EVT-* / STP-NOT-* | Publish CloudEvent through outbox; send approved notification. |

| 17 | Response | STP-RSP-* | Build success response, safe error response, or async acknowledgement. |

| E | Error Path | STP-ERR-* / STP-CMP-* | Classify error, retry, compensate, DLQ, escalate, safe failure response. |

| Template | Use When | Required Additional Controls |

| --- | --- | --- |

| KAFKA_ASYNC | Event-driven work can complete independently. | CloudEvents/Avro schema, consumer idempotency, DLQ, replay procedure. |

| TEMPORAL_MACHINE | Long-running machine workflow with retries, sagas, or compensation. | Determinism, version pinning, activity timeouts, compensation tests. |

| AI_RAG_QUERY | Retrieval, summarization, extraction, recommendation, or model answer is needed. | Input/retrieval/execution/output guardrails, LiteLLM route, citations, audit. |

| DOCUMENT_INTAKE | File upload, malware scan, OCR, metadata, embeddings, or OpenKM processing. | Checksum, AV scan, OpenKM reference, OCR confidence, lineage, reprocess. |

| RECONCILIATION | Repair, reprocess, replay, rebuild index/cache, or correct drift. | Human approval if state changing, replay limits, before/after evidence. |

| RETENTION | Archive, legal hold, purge, supersede, or retirement action. | Retention authority, legal hold check, irreversible action approval. |

# 7. Business MicroFunction Extension Pattern

The only normal coding path is to create an isolated, reusable STP-BUS-* function when existing standard steps, rules, DMN, OPA policies, or configuration cannot satisfy the business requirement.

| Design Element | Mandatory Rule |

| --- | --- |

| Name | Use STP-BUS-<BOUNDED-CONTEXT>-<ACTION>-<NNN>, e.g., STP-BUS-LOAN-CHECK-ELIGIBILITY-001. |

| Ownership | Assign one bounded-context owner and one technical owner. No orphan functions. |

| Input | Use typed input context or validated parameter object. Do not parse raw HTTP/Kafka payloads inside business logic. |

| Output | Return typed result, domain decision, state transition request, or business event intent. Do not build HTTP responses. |

| Dependencies | Use injected ports and domain services. Do not depend directly on DB clients, Kafka producers, Redis clients, model SDKs, or OpenKM clients. |

| Errors | Raise typed domain or technical errors and let the envelope classify, retry, compensate, escalate, or respond safely. |

| Idempotency | Declare whether the function is read-only, idempotent mutation, or non-idempotent and must be protected by an outer idempotency step. |

| Observability | Emit domain-safe metrics and structured events through the framework. No PII or secrets in logs. |

| Tests | Cover happy path, invalid input, boundary conditions, authorization/classification assumptions, idempotency, retry behavior, error mapping, and compensation where applicable. |

| Catalog | Register function in step_catalog with owner, version, classification ceiling, capability, default policies, and status. |

Appendix Preview: STP-BUS-* class-level pattern

| // Reference pattern only - exact package and framework names are governed by 10B.  
public final class LoanEligibilityMicroFunction  
        implements MicroFunction<LoanEligibilityInput, LoanEligibilityResult> {  
  
    private final LoanPolicyPort loanPolicyPort;  
    private final CustomerProfilePort customerProfilePort;  
  
    public LoanEligibilityMicroFunction(  
            LoanPolicyPort loanPolicyPort,  
            CustomerProfilePort customerProfilePort) {  
        this.loanPolicyPort = loanPolicyPort;  
        this.customerProfilePort = customerProfilePort;  
    }  
  
    @Override  
    public MicroFunctionResult<LoanEligibilityResult> execute(  
            ExecutionContext context,  
            LoanEligibilityInput input) {  
        // Business rule only. No HTTP response, no direct DB, no Kafka, no model call.  
        var profile = customerProfilePort.findProfile(input.customerId(), context.tenantId());  
        var decision = loanPolicyPort.evaluate(profile, input.loanRequest());  
        return MicroFunctionResult.success(new LoanEligibilityResult(decision));  
    }  
} |

## 7.1 Design Review Checklist for New Business Functions

Is the function truly required, or can the behavior be configured using standard steps, DMN, OPA, or parameters?

Does the function belong to exactly one bounded context?

Does the function have one responsibility and a typed contract?

Does the function depend only on approved ports, not infrastructure adapters?

Are all errors typed and delegated to framework error policy?

Is idempotency declared and tested?

Are audit, observability, security, and safe responses framework-managed?

Are tests deterministic, synthetic-data-only, and linked to requirements?

Is the catalog registration complete with owner, version, classification ceiling, policies, and status?

# 8. Error Handling, Compensation, and Safe Response Design

Error handling is a standard framework concern, not a business-function afterthought. Every step must have a known failure behavior before activation.

| Failure Type | Required Design Response | Developer Responsibility |

| --- | --- | --- |

| Validation failure | Return safe problem response; do not execute business or persistence steps. | Provide clear validation rule and synthetic tests. |

| Authorization or classification failure | Fail closed, audit denial, do not reveal sensitive details. | Never catch and bypass security failure. |

| Transient dependency failure | Apply configured timeout, retry, circuit breaker, fallback, and DLQ where applicable. | Declare retryability and ensure idempotency. |

| Business rule rejection | Return domain-safe rejection reason and audit decision evidence. | Use typed domain result; no stack traces. |

| Partial mutation | Compensate or reconcile through saga-safe action. | Declare compensation step and test failure path. |

| AI/model/guardrail failure | Block or degrade safely; route to human review if high impact. | Do not bypass LiteLLM, guardrails, OPA, or Harness. |

| Unknown exception | Classify as safe internal error, audit, alert, and generate DLQ/escalation where applicable. | Never swallow exception or return misleading success. |

## 8.1 Compensation and Reversibility Rules

A mutating step must declare whether it is naturally reversible, compensatable, forward-fix only, or irreversible requiring human approval.

A compensation step must be tested using synthetic failure injection before activation.

The framework must persist enough evidence to reconstruct before-state, after-state, actor, policy, trace, configuration version, and compensation result.

Rollback must not mean ungoverned direct database edits. Use forward-fix migration, compensating transaction, deactivation, cache invalidation, or replay under approval.

# 9. Data, Integration, Security, and AI Design Rules

| Area | Design Rule |

| --- | --- |

| Data Access | Business functions use repositories/ports approved by the bounded context. Free-form SQL in configuration and cross-context writes are prohibited. |

| Integration | Outbound calls use STP-INT-* adapters with timeout, retry, circuit breaker, classification handling, schema validation, and audit. |

| Events | Events use CloudEvents and schema governance. Business functions return event intent; transactional outbox publishes through the framework. |

| Cache | Caffeine/Redis are acceleration layers only. Cache loss must degrade latency, not correctness. |

| Security | Identity, authorization, classification, SBAC, OPA, and secret handling are mandatory framework controls. |

| AI Retrieval | Retrieval functions must carry source citations, classification eligibility, freshness checks, conflict flags, and confidence metadata. |

| AI Model Calls | All model calls use LiteLLM aliases. Direct model-provider SDK calls are prohibited. |

| Tool Actions | AI-proposed tool actions go through NeMo Execution Rail, Harness, SBAC, OPA, trust scoring, audit, and human approval when required. |

## 9.1 Classification-Aware Design

| Classification | Design Handling |

| --- | --- |

| Public | May use approved public/cloud paths if contract, audit, and model-route rules are satisfied. |

| Internal | May use approved corporate cloud routes with workspace controls and no secrets/PII leakage. |

| Confidential | Requires stronger redaction, access control, audit, and approved model route; human review for high-impact outputs. |

| Restricted | Must use approved private/on-prem route unless a formal waiver exists; no uncontrolled prompt upload or derivative exposure. |

# 10. Testing and Architecture Fitness Functions

A MicroFunction is not complete when it compiles. It is complete when its behavior, architecture boundary, error behavior, security behavior, and evidence are test-proven.

| Fitness Function | What It Proves | Preferred Enforcement |

| --- | --- | --- |

| SOLID and package boundary | Domain/business functions do not depend on controllers, adapters, DB clients, Kafka, model SDKs, or framework internals. | ArchUnit, Semgrep, import lint, CLAUDE.md checks. |

| Clean Architecture boundary | Dependencies point inward; adapters implement ports; business logic is infrastructure-independent. | ArchUnit and code review checklist. |

| DDD ownership | Function belongs to one bounded context and does not write another context's schema. | Repository rules, contract tests, database ownership checks. |

| Idempotency | Retry or replay does not duplicate business effect. | Unit/component tests, idempotency registry tests. |

| Determinism | Time, randomness, network, and external dependency behavior are controlled. | Fixed clocks, seeded randomness, mocked ports. |

| Error and compensation | Expected failures route to retry, safe response, DLQ, human review, or compensation. | Failure injection tests and compensation tests. |

| Security and classification | Unauthorized, unskilled, or ineligible data/model route is blocked. | OPA tests, SBAC tests, guardrail tests. |

| Observability | Trace, metrics, logs, and audit evidence are emitted and redacted. | Component tests and evidence schema validation. |

| Contract compatibility | Input/output, error shape, events, and schema do not break consumers. | OpenAPI/AsyncAPI/schema compatibility tests. |

| Regression and mutation | Tests detect meaningful behavioral changes. | Coverage, mutation score, changed-line coverage. |

## 10.1 Minimum Test Set for STP-BUS-*

Happy path proves the expected business outcome.

Invalid input, boundary input, missing required data, and malformed context are rejected safely.

Authorization/classification assumptions are enforced before business execution.

Idempotency or read-only behavior is proven.

Downstream port failure produces the configured error classification and safe response.

Compensation or reconciliation path is tested for mutating operations.

No real customer data, production data, credentials, or PII appears in test fixtures.

Test evidence links to ticket, function code, configuration version, MR, and AVCI summary.

## 11.1 Progressive Autonomy Rules

Level 0: AI drafts analysis only; no repository change.

Level 1: AI proposes configuration or code changes in a branch; human executes review and merge.

Level 2: AI may run approved non-production tests through Harness after SBAC/OPA approval.

Level 3: AI may open remediation PRs with evidence, but cannot approve, merge, deploy, or override policy.

Level 4: Limited operational action may be delegated only for low-risk, reversible, pre-approved actions with audit and rollback evidence.

Production-impacting, destructive, Confidential/Restricted, or policy-exception changes require named human approval.
