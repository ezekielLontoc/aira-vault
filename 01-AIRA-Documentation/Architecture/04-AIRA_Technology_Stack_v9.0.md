# 1. Executive Summary

This v9.0 revision updates the AIRA Technology Stack after the adoption of the Enterprise Design Principles and SOLID Enforcement Layer. It preserves the v8.0 position that technology is not a simple inventory; it is an enforceable control baseline for architecture discipline, secure-by-design implementation, reproducible delivery, governed AI acceleration, and evidence by construction.

The primary v9.0 changes are the controlled addition of Dograh as the Voice Agent Orchestration Platform and GitNexus as a governed code-intelligence and AI-agent context tool. Dograh is not treated as the only STT or TTS engine. Whisper, faster-whisper, whisper.cpp, Piper, Kokoro, and other approved providers remain swappable provider adapters behind AIRA controls. GitNexus is not a source-control replacement and must not execute changes; it is a derivative code-intelligence layer for impact analysis, dependency graphing, call-chain understanding, and MCP-mediated context delivery to approved AI assistants.

| v9.0 Non-Negotiable Technology Rule  
No AIRA technology, library, framework, model, voice component, code-intelligence component, plug-in, build tool, runtime, database extension, workflow component, prompt tool, observability component, or agent connector may be adopted unless it preserves or improves SOLID compliance, Clean Architecture boundaries, DDD bounded contexts, testability, security posture, observability, reversibility, and AVCI evidence. |

| Controlled improvement | Auto-Heal, Auto-Learn, and Auto-Improve may propose technology refreshes only through reviewable, tested, reversible, evidence-producing paths. |

# 3. Technology Governance Principles

| ID | Principle | Technology Governance Meaning |

| --- | --- | --- |

| TG-01 | AVCI compliance | Every component has owner, source, version, validation evidence, classification, usage scope, and improvement path. |

| TG-02 | Enterprise principles as adoption gate | SOLID, Clean Architecture, DDD, ports-and-adapters, DRY/KISS/YAGNI, idempotency, determinism, fail-safe behavior, human oversight, least privilege, separation of duties, observability, policy as code, testability, security, resilience, fitness functions, progressive autonomy, reversibility, and AVCI are mandatory decision criteria. |

| TG-03 | Model gateway first | All model traffic routes through LiteLLM aliases. Direct provider SDK calls from applications, scripts, notebooks, Dograh flows, GitNexus-connected agents, or task agents are prohibited. |

| TG-04 | Guardrails always | Input, Retrieval, Execution, and Output checkpoints are mandatory for AI and voice-AI flows. Protected operations fail closed when guardrails are unavailable. |

| TG-05 | Hermes reasons; Harness executes | AI reasoning and execution are separated. Harness, SBAC, OPA, trust scoring, and audit govern tool actions. GitNexus may inform; it must not execute. |

| TG-06 | Voice orchestration is not model authority | Dograh orchestrates voice interaction, but LLM route, STT/TTS provider choice, classification, transcript retention, and tool action remain governed by AIRA. |

| TG-07 | GitOps and policy-as-code | Git is source of truth for infrastructure, manifests, policies, prompts, guardrails, model aliases, voice-flow configuration, and pipelines. |

| TG-08 | Secure by design | Identity, secrets, workload identity, scanning, logging redaction, tenant isolation, classification handling, transcript protection, and repo-index protection are stack requirements. |

| TG-09 | Observable by default | Services, workflows, voice interactions, AI interactions, model routing, GitNexus context retrieval, and platform actions emit metrics, logs, traces, audit, and evidence references. |

| TG-10 | No deprecated foundations for greenfield | AIRA will not start greenfield development on deprecated, EOL, non-supported, or non-approved foundations without formal waiver and exit plan. |

# 5. Technology Status Definitions

| Status | Meaning | Approval / Control Requirement |

| --- | --- | --- |

| Mandatory | Required for production unless formal waiver exists. | Architecture, Security, and Change approval required to bypass. |

| Primary | Default choice for new AIRA development. | Alternative requires ADR/TDL and impact assessment. |

| Approved | Permitted where appropriate. | Must follow configuration, security, monitoring, and evidence rules. |

| Conditional | Allowed only after readiness, compatibility, licensing, security, or cost review. | Waiver or ADR required before production use. |

| Evaluation | Allowed only in PoC, sandbox, or architecture lab. | No production data or production workflow dependency. |

| Transition / Watch | Supported temporarily while upgrade, migration, or compatibility work is completed. | Time-bounded plan and exit criteria required. |

| Deprecated | Existing use only; no new deployment. | Migration or retirement plan required. |

| Not Approved | Prohibited for AIRA code, data, production workflows, or AI-assisted development. | Use creates non-conformance and must be removed. |

# 7. Dograh Voice Agent Orchestration Governance

Dograh is added as the AIRA Voice Agent Orchestration Platform under Conditional / Controlled Evaluation status. It may improve AIRA by providing a self-hostable voice-agent workflow layer for inbound/outbound calling, telephony/WebRTC integration, and conversational flow design. | Control Area | Mandatory Dograh Governance Rule | Evidence Required |

# 8. GitNexus Code Intelligence Governance

GitNexus is added as a governed code-intelligence layer for repository indexing, dependency/call graph creation, blast-radius analysis, and AI-agent codebase context. It can materially improve AI-assisted development and Auto-Heal/Auto-Improve analysis by allowing assistants to reason across code relationships. It must remain read-only, derivative, and non-authoritative. It does not replace Git, CODEOWNERS, ArchUnit, SonarQube, Semgrep, LLM Wiki, ADRs, API contracts, or human code review.

| Control Area | Mandatory GitNexus Governance Rule | Evidence Required |

| --- | --- | --- |

| Technology Role | GitNexus provides code-intelligence context, repo graphing, call-chain analysis, and MCP-mediated assistance. | Tool registry entry, owner, source, version, architecture decision. |

| Read-Only Boundary | GitNexus must not commit, merge, approve, deploy, edit production config, or execute tools. | Read-only token, network policy, MCP tool permissions, blocked write-action test. |

| Source Authority | SCM, contracts, tests, ADRs, and CI remain authoritative. GitNexus index is derivative. | Index provenance, source commit hash, rebuild procedure, freshness evidence. |

| Secret Safety | Repositories must be secret-scanned before indexing; indexes must exclude secrets, credentials, production data, and restricted artifacts. | Gitleaks result, exclusion config, DLP test, index retention rule. |

| Classification | Only approved repositories and branches may be indexed. Confidential/Restricted code requires private/on-prem processing. | Classification record, ACL test, storage encryption, retention approval. |

| AI Assistant Integration | MCP/CLI/browser context must pass through approved assistant governance and CLAUDE.md rules. | MCP registry, tool scope, prompt log redaction, AI usage declaration. |

| Auto-Heal / Improve | GitNexus may generate impact analysis and test recommendations only. Fixes remain PR/MR candidates. | Candidate PR, affected-test list, architecture impact evidence, human checker. |

| Observability | GitNexus context retrieval, index refresh, and agent access are logged and traceable. | Tool access log, trace_id, repo_id, commit_sha, user/agent identity. |

| Security Review | Pin to approved release/fork; scan dependencies and runtime; no uncontrolled external callbacks. | SBOM, SCA/SAST result, egress policy, signed artifact. |

| Rebuild / Retirement | Indexes are rebuildable from source and retired on repo/archive/branch deletion or supersession. | Rebuild job, retention policy, stale index quarantine evidence. |

| GitNexus Adoption Verdict  
Approved for controlled developer and architecture-review evaluation. It should be used as a codebase-understanding accelerator and CI/review evidence contributor, not as an autonomous engineering actor. |

# 9. Architecture Fitness Function Tooling Baseline

Architecture fitness functions are technology-backed tests that continuously verify that AIRA remains aligned with its intended architecture. They are mandatory for high-risk repositories and must run locally where feasible and in CI strict mode before merge.

| Fitness Area | Recommended Tooling | Minimum Rule |

| --- | --- | --- |

| Layer dependency direction | ArchUnit, dependency graph checks, GitNexus graph review | Domain must not depend on infrastructure/adapters; controllers must not contain business logic. |

| Package and bounded-context ownership | ArchUnit, CODEOWNERS, repository rules, GitNexus | No cross-context internal package access or shared schema mutation without ADR. |

| Direct model provider prevention | Semgrep, import lint, CLAUDE.md hook | Provider SDK imports and direct provider endpoints are blocked. |

| Voice provider independence | Dograh adapters, contract tests, provider registry | Dograh flows cannot hardcode one STT/TTS/LLM provider outside approved config. |

| Policy-as-code validation | OPA/Rego tests, Conftest, Gatekeeper/Kyverno | Policy bundles and manifests must pass deterministic tests. |

| Contract compatibility | OpenAPI/AsyncAPI diff, Spring Cloud Contract, Pact where approved | Breaking API/event changes require versioning and deprecation plan. |

| Test sufficiency | JUnit/Vitest/pytest coverage, mutation testing, changed-line coverage | High-risk code requires branch and mutation evidence, not line coverage alone. |

| Security regression | Semgrep, OWASP ZAP, Trivy, Gitleaks | Security defects, secret leaks, unsafe logging, and vulnerable images block merge. |

| Reversibility | Flyway checks, Argo CD sync plan, Helm diff, feature flag plan | Production-impacting changes require rollback, forward-fix, or compensation path. |

| Observability completeness | OTel instrumentation checks, log schema validation, dashboard/alert evidence | New critical flows emit trace, metric, log, audit, and evidence references. |

| Code intelligence freshness | GitNexus index timestamp/commit check | AI assistants must not use stale code graphs as current authority. |

# 16. AVCI Compliance Summary

| AVCI Property | How v9.0 Satisfies It |

| --- | --- |

| Attributable | Every technology has owner, source, version, status, decision record, accountable role, and source-of-truth path. Dograh and GitNexus have explicit owners and control boundaries. |

| Verifiable | Each component requires tests, scans, fitness checks, compatibility evidence, supply-chain evidence, restore/rollback proof, and runtime observability. Voice and code-intelligence pilots have dedicated verification gates. |

| Classifiable | Technology use is governed by data classification, model-routing eligibility, logging/retention rules, transcript handling, repository-index handling, and access-control scope. |

| Improvable | Quarterly refresh, dependency automation, CVE response, incident learning, Dograh pilot findings, GitNexus pilot findings, and governed Auto-Improve proposals provide controlled improvement. |

| Final v9.0 Technology Adoption Rule  
AIRA adopts technology for controlled capability, not novelty. Dograh may improve voice-agent orchestration only if it preserves provider independence, classification, guardrails, and evidence. GitNexus may improve codebase understanding only if it remains read-only, derivative, secret-safe, and governed. Any technology that accelerates delivery while weakening architecture, security, testability, observability, reversibility, or AVCI evidence is rejected or quarantined until remediated. |
