# 1. Executive Summary

AIRA is a regulated, AI-native enterprise platform. Security must be embedded into identity, authorization, secrets, workload identity, service-to-service trust, AI agent execution, logging, evidence, incident response, and operational recovery from the first line of code.

Version 2.0 upgrades the security baseline by incorporating the Enterprise Design Principles and SOLID Enforcement Layer. Security controls must preserve Clean Architecture boundaries, DDD ownership, ports-and-adapters separation, least privilege, separation of duties, fail-closed behavior, testability, observability, reversibility, and AVCI evidence.

| Security Area | AIRA v2.0 Standard Position |

| --- | --- |

| Identity Provider | Keycloak federated to corporate Active Directory / enterprise IdP. No local application passwords except controlled break-glass accounts. |

| Authentication | OIDC/OAuth 2.x patterns with MFA, PKCE where applicable, short-lived tokens, token audience validation, and session evidence. |

| Authorization | RBAC for coarse roles, ABAC for contextual conditions, SBAC for human/AI skills, and OPA for policy-as-code decisions. |

| Secrets | Vault or approved enterprise secrets manager controls dynamic/leased secrets, PKI, encryption material, and secret access evidence. |

| Workload Identity | SPIFFE IDs and SVIDs for service identity; mTLS for service-to-service communication; static shared credentials rejected. |

| AI Agent Execution | Hermes may reason; Harness executes; OPA decides; SBAC and trust score gate actions; humans approve high-impact actions. |

| Audit Evidence | Every access decision records actor, subject, resource, action, decision, policy version, classification, and trace_id. |

| Security Architecture Rule<br>AI may recommend, draft, analyze, or classify security work. It must not approve access, bypass policy, expose secrets, execute production-impacting actions, suppress evidence, or weaken security posture. |

# 3. Non-Negotiable Security Principles

| ID | Principle | Engineering Meaning |

| --- | --- | --- |

| SEC-01 | Zero Trust by Default | No user, service, agent, device, network segment, or environment is trusted implicitly. Every request is authenticated, authorized, classified, logged, and trace-correlated. |

| SEC-02 | Named Accountability | Human access is tied to one named person. Shared accounts are prohibited except approved break-glass with dual approval and post-use review. |

| SEC-03 | Least Privilege and Just-in-Time | Users, services, agents, and pipelines receive only the authority required, for the shortest viable time, with revocation evidence. |

| SEC-04 | Policy as Code | Authorization rules, admission controls, access decisions, model-routing rules, and guardrail approvals are reviewed policy artifacts. |

| SEC-05 | No Secret Exposure | Secrets never appear in Git, prompts, wiki pages, logs, traces, errors, screenshots, test fixtures, or documents. |

| SEC-06 | Workload Identity over Static Credentials | Service identity uses SPIFFE/SVID or approved short-lived workload credentials. Static credentials require waiver and rotation evidence. |

| SEC-07 | Classification Drives Controls | Public, Internal, Confidential, and Restricted classifications determine routing, encryption, logging, retention, approval, and AI use. |

| SEC-08 | AI Actions Are Mediated | AI agents cannot directly execute tools. Harness mediates identity, skill, trust, policy, scope, approval, and audit. |

| SEC-09 | Fail Closed | If OPA, Vault, Keycloak, Harness, NeMo Guardrails, LiteLLM, or audit evidence pipeline is unavailable, protected actions are blocked. |

| SEC-10 | AVCI Evidence on Every Control | Every access decision and security control must be attributable, verifiable, classifiable, and improvable. |

# 4. Reference Security Architecture

AIRA uses layered enforcement. Gateway controls do not replace service controls; service controls do not replace data controls; AI guardrails do not replace OPA; and OPA decisions do not replace human accountability for high-impact actions.

| Layer | Primary Controls | Mandatory Evidence |

| --- | --- | --- |

| Identity Layer | Keycloak, corporate AD/IdP, MFA, group/role mapping, device posture where available. | User ID, realm, groups, MFA status, token issuer, login event, revocation event. |

| Gateway Layer | TLS, JWT validation, audience validation, rate limiting, WAF/API policies, trace_id injection. | Request ID, actor, tenant, route, decision, rate-limit result, security headers. |

| Service Layer | Spring Security, resource authorization, OPA client, MicroFunction envelope, method security. | Subject, action, resource, policy version, classification, decision, trace_id. |

| Data Layer | RLS, schema ownership, encryption, audit columns, least-privilege DB roles. | Tenant_id, actor_id, transaction_id, mutation history, migration version. |

| Secrets Layer | Vault dynamic secrets, leases, rotation, PKI, transit encryption, sealed secret workflow. | Secret path, lease ID, TTL, accessor, rotation event, revocation event. |

| Workload Layer | SPIFFE IDs, SVIDs, mTLS, service mesh/network policy, short-lived certificates. | SPIFFE ID, certificate serial, trust domain, workload attestation event. |

| AI Execution Layer | Harness, SBAC, OPA, trust scoring, NeMo Guardrails, LiteLLM routing. | Agent ID, skill, trust score, model alias, guardrail result, approval, action result. |

# 5. Enterprise Design Principles and SOLID Enforcement for Security

Security implementation must preserve the 20 Enterprise Design Principles. Security features are not exempt from SOLID or Clean Architecture; they must be modular, testable, observable, and reversible like any other production capability.

| Principle Group | Security Enforcement Meaning |

| --- | --- |

| SOLID | Security services, authorization evaluators, secret adapters, token validators, and audit writers must have single responsibility, stable contracts, substitutable implementations, segregated interfaces, and dependency inversion. |

| Clean Architecture | Domain logic must not depend directly on Keycloak, Vault, OPA, Kafka, database drivers, or model providers. Security dependencies enter through ports/adapters. |

| DDD / Bounded Contexts | Business roles, permissions, data classification, and ownership are defined within bounded contexts; cross-context privilege expansion requires ADR or approved policy. |

| Ports and Adapters | Identity, policy, secrets, workload identity, audit, and notification integrations are adapters behind ports; provider-specific code does not leak into domain services. |

| DRY, KISS, YAGNI | No duplicated custom authorization logic, secret-loading logic, or error-mapping logic in each service. Use framework controls unless justified. |

| Idempotency / Determinism | Access grants, secret rotations, policy bundles, and remediation actions must be retry-safe and produce reproducible evidence. |

| Fail-Safe / Human-in-the-Loop | High-impact and low-confidence actions require human approval. Protected operations block when controls are unavailable. |

| Least Privilege / SBAC / Separation of Duties | Authority is scoped by role, context, skill, trust, and task; requester, approver, executor, and reviewer duties are separated. |

| Observability / Policy as Code / Testability | Security decisions are logged, metrics-ready, policy-tested, CI-gated, and verifiable through architecture fitness functions. |

| Secure by Design / Resilience / Reversibility / AVCI | Security is embedded in design, resilient to dependency failure, reversible through revocation and rollback where possible, and evidenced under AVCI. |

| Security SOLID Rule<br>No security feature may be accepted if it introduces hidden coupling, duplicated privilege logic, direct provider dependency in domain code, untestable policy behavior, unobservable access decisions, or irreversible privilege expansion. |

# 7. Authorization Standard: RBAC, ABAC, SBAC, and OPA

AIRA uses layered authorization. RBAC grants coarse role membership, ABAC evaluates contextual attributes, SBAC gates skill-based actions for humans and agents, and OPA provides policy-as-code decisions. Application code must not embed scattered hardcoded authorization decisions.

| Authorization Layer | Use | Reject Pattern |

| --- | --- | --- |

| RBAC | Coarse human and service roles such as Developer, Approver, DBA, Security Admin, AI Agent Owner. | Role explosion, shared admin role, direct production role by default. |

| ABAC | Tenant, branch, classification, environment, data domain, time, device posture, risk score, and workflow status. | Authorization based only on UI hiding or frontend checks. |

| SBAC | Skill-gated actions for humans and AI agents: code review, remediation, security triage, data migration, model route change. | AI agent with unrestricted tool scope or unverified skill. |

| OPA | Central policy decisions for APIs, tools, CI gates, Kubernetes admission, Harness actions, secrets, and model routing. | Hardcoded if/else policy logic in business code. |

| Human Approval | Required for high-risk, Confidential/Restricted, destructive, production-impacting, or low-confidence actions. | Auto-approve production changes or self-approval. |

# 8. Secrets, Keys, Certificates, and Vault Governance

| Control | Standard | Evidence / Gate |

| --- | --- | --- |

| Secrets Source | Vault or approved enterprise secrets manager is authoritative. Git, CI variables, prompts, and documentation are not secret stores. | Vault path ownership, lease record, rotation evidence. |

| Dynamic Secrets | Use leased/dynamic database and service credentials where supported. Static secrets require exception. | Lease TTL, revocation test, rotation schedule. |

| Secret Injection | Secrets are mounted or injected at runtime through approved mechanisms; never baked into images or committed to repositories. | Image scan, secret scan, deployment manifest review. |

| Encryption Keys | Keys are owner-assigned, versioned, rotated, access-controlled, and separated by environment and classification. | Key inventory, rotation record, access review. |

| PKI and Certificates | Short-lived internal certificates use Vault PKI, SPIFFE/SVID, step-ca, or enterprise CA. Manual cert sprawl is prohibited. | Certificate inventory, expiry alert, issuance audit. |

| Emergency Access | Emergency secret disclosure requires dual approval, time-boxed access, ticket, post-use rotation, and RCA. | Break-glass ticket, approvers, rotation proof. |

| No Secret Exposure Rule<br>Any secret detected in Git, prompt, log, trace, screenshot, test fixture, wiki page, or document is a security incident and must trigger revocation, rotation, evidence preservation, and post-incident improvement. |

# 9. Workload Identity, mTLS, and Service-to-Service Trust

Services must authenticate to each other using workload identity, not static shared secrets. SPIFFE IDs and SVIDs are the preferred pattern for service identity and mTLS trust.

| Area | Mandatory Requirement |

| --- | --- |

| SPIFFE ID Naming | Each service receives a stable SPIFFE ID tied to environment, namespace, service name, and trust domain. |

| mTLS | Service-to-service communication must use mTLS where feasible. Plain internal HTTP is not acceptable for protected data paths. |

| Service Authorization | A valid certificate authenticates the workload; OPA/service policy still authorizes the action. |

| Certificate Rotation | Certificates are short-lived and rotated automatically. Expiry alerts must be observable. |

| Network Policy | Kubernetes network policies and ingress/egress rules restrict communication to approved paths. |

| Evidence | Workload attestation, certificate serial, policy decision, and trace_id are retained for audit. |

# 10. Classification, Data Handling, Logging, and Model Routing

| Classification | Security Handling | AI / Model Routing |

| --- | --- | --- |

| Public | Standard integrity checks, normal logging without secrets, approved public examples. | Approved cloud routes permitted if source and output are non-sensitive. |

| Internal | Authenticated access, normal enterprise retention, redacted logs. | Approved workspace/cloud route may be permitted subject to policy. |

| Confidential | Least privilege, masking/redaction, stronger audit, restricted retrieval, explicit sharing control. | Private or approved enterprise route only; no uncontrolled prompt upload. |

| Restricted | Strongest access control, no raw exposure in logs/prompts/tests, human approval for high-impact use. | On-prem/private model route required unless formal waiver is approved. |

| Prohibited Data Handling Pattern | Reason |

| --- | --- |

| Raw PII or account data in logs, prompts, test fixtures, cache keys, traces, or screenshots. | Creates privacy, leakage, and uncontrolled retrieval risk. |

| Secrets in examples, documentation, or wiki articles. | Documentation can be indexed and retrieved by AI tools. |

| Classification omitted from API, event, data, prompt, or audit artifacts. | Routing, retention, logging, and access decisions cannot be enforced. |

| Direct model-provider call using provider SDK or hardcoded model name. | Bypasses LiteLLM, guardrails, routing, budgets, audit, and classification policy. |

# 12. AI Agent Execution Security and Harness Boundary

AI agent execution is a security-controlled workflow. An agent may reason and propose; it cannot directly execute Git, CI/CD, infrastructure, database, document store, model-routing, production API, or secrets actions.

| Step | Required Control |

| --- | --- |

| 1. Task Intake | Human/system request is classified, scoped, and trace-correlated. |

| 2. Evidence Assembly | Agent retrieves only classification-eligible, approved, current sources. |

| 3. Guardrail Evaluation | Input, Retrieval, Execution, and Output guardrails are applied as appropriate. |

| 4. SBAC Check | Agent/human must hold active skill at required level with valid dependencies. |

| 5. Trust Score Check | Trust score must meet action risk threshold and decay rules. |

| 6. OPA Decision | Policy permits, denies, or requires human approval. |

| 7. Harness Execution | Approved tool action is executed only through Harness using mediated credentials. |

| 8. Evidence Capture | Action result, policy decision, approval, artifacts, and trace_id are written to audit. |

| 9. Human Review | Production, destructive, Confidential/Restricted, or low-confidence changes require named human approval. |

| Explicitly Rejected AI Patterns<br>Direct agent credentials, provider SDK bypass, guardrail bypass flags, autonomous production execution, direct database edits, direct kubectl writes, unreviewed prompt/guardrail updates, and AI-approved access grants are rejected security patterns. |

# Appendix C. Security PR/MR Compliance Summary

| Security / AVCI Compliance Summary  
- Security impact: None / Low / Medium / High / Critical  
- Identity impact: roles, claims, scopes, sessions, MFA, service account changes  
- Authorization impact: RBAC / ABAC / SBAC / OPA policy references  
- Secrets impact: Vault path, rotation evidence, secret scan result  
- Classification impact: Public / Internal / Confidential / Restricted  
- SOLID / architecture boundary impact: preserved / improved / exception requested  
- Test evidence: security tests, OPA tests, fail-closed tests, boundary tests
