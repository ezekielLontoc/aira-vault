# 1. Executive Summary

This guide converts the AIRA DevSecOps operating model into an executable CI/CD implementation baseline. It defines how every code, configuration, prompt, guardrail, infrastructure, database, MicroFunction, and documentation change moves from developer workstation to controlled runtime through verifiable pipeline gates.

The guide is intentionally implementation-oriented. It provides the stage model, evidence pack structure, quality gates, architecture fitness functions, policy-as-code checks, signed provenance requirements, and copy-ready pipeline templates that the software development and DevSecOps teams can use during Sprint-0 and foundation build.

| Strategic Outcome | How This Guide Delivers It |

| --- | --- |

| Evidence by construction | The pipeline generates a standard evidence pack containing test, security, architecture, AI, supply-chain, deployment, and approval evidence. |

| SOLID and architecture protection | Architecture fitness functions reject boundary violations, direct provider calls, cross-context writes, dependency inversions, and bypass patterns. |

| Secure release governance | SAST, SCA, secret scanning, SBOM, image scanning, DAST, OPA, Conftest, signature, and provenance gates are mandatory. |

| AI-safe engineering | Codex/ChatGPT/agent involvement is declared in PR evidence, evaluated through guardrails, and restricted by Harness, SBAC, OPA, and human review. |

| Reversibility and auditability | Every promotion requires rollback or forward-fix plan, evidence manifest, change ticket, approver, and retained artifacts. |

Core rule: the pipeline is allowed to reject a change even when the code compiles. AIRA production readiness requires complete evidence, not only successful execution.

# 2. Purpose, Scope, and Authority

## 2.1 Purpose

Define the standard CI/CD stage model for all AIRA repositories.

Define mandatory quality, security, architecture, AI, supply-chain, and evidence gates.

Provide a reusable evidence pack schema that satisfies AVCI.

Provide copy-ready GitHub Actions, GitLab CI, pre-commit, and evidence manifest templates.

Define the promotion controls from DEV to INT, UAT, STG, and PROD.

## 2.2 In Scope

Application services, frontend applications, MicroFunction framework repositories, infrastructure repositories, database migration repositories, prompt/guardrail repositories, and documentation repositories.

All human-authored, AI-assisted, and automation-generated changes.

All pipeline evidence used for internal audit, compliance, security review, release approval, and post-incident reconstruction.

## 2.3 Out of Scope

Detailed production infrastructure installation steps, which belong to the Runtime Platform Bootstrap Runbook.

Business UAT sign-off procedures beyond the pipeline evidence interface.

Full DR/BCP playbooks, except rollback and release evidence requirements defined here.

## 2.4 Authority

This guide is subordinate to the AIRA Architecture Blueprint, AVCI Engineering Standard, Enterprise Design Principles and SOLID Enforcement Layer, Technology Stack, DevSecOps Standard, CLAUDE.md Reference, Unit Testing Standard, and Repository Bootstrap Guide. If pipeline behavior conflicts with an approved higher-level standard, the higher-level standard governs and this guide must be updated through ADR/TDL change control.

# 3. CI/CD Governance Principles

| ID | Principle | Implementation Meaning |

| --- | --- | --- |

| CICD-01 | Git is the source of truth | All pipeline, policy, prompt, guardrail, schema, and infrastructure changes originate from reviewed Git changes. |

| CICD-02 | Evidence before promotion | A change cannot be promoted without a complete evidence pack. |

| CICD-03 | Fail safe, not fail open | If a required gate, scanner, policy decision, signer, audit writer, or evidence store is unavailable, the pipeline fails. |

| CICD-04 | Architecture is tested | SOLID, Clean Architecture, DDD, ports/adapters, no direct provider calls, and bounded-context rules are enforced by automated checks. |

| CICD-05 | Security is a gate | Secrets, vulnerabilities, unsafe dependencies, unsafe container images, and policy violations block merge or promotion. |

| CICD-06 | AI involvement is declared | AI-assisted changes disclose tool/model usage, prompt intent, files affected, tests run, and human checker. |

| CICD-07 | Reproducible builds | Builds use pinned versions, dependency locks, clean runners, and signed artifacts. |

| CICD-08 | Promotion is deliberate | DEV, INT, UAT, STG, and PROD promotion gates have explicit evidence, approval, and rollback criteria. |

# 4. Target Pipeline Architecture

The standard AIRA pipeline separates source validation, build, test, security, architecture validation, evidence generation, artifact signing, and promotion. This prevents a single successful build from being mistaken for full production readiness.

Figure 1. AIRA CI/CD target pipeline architecture.

## 4.1 Pipeline Layers

| Layer | Responsibility |

| --- | --- |

| Developer Workstation | Pre-commit, local tests, secret scanning, formatting, CLAUDE.md rule checks, devcontainer controls. |

| Source Control | Branch protection, signed commits, PR/MR templates, CODEOWNERS, required checks, approval rules. |

| CI Validation | Build, test, SAST, SCA, SBOM, architecture fitness, contract tests, AI artifact tests. |

| Artifact and Evidence | OCI images, packages, SBOM, provenance, signatures, evidence manifests, scan results. |

| CD Promotion | Environment-specific deployment, GitOps reconciliation, smoke tests, DAST, change approval, rollback/forward-fix plan. |

| Audit and Knowledge | Evidence retention, OpenKM record, LLM Wiki/Obsidian approved summaries, Auto-Learn candidate creation. |

# 5. Stage Model and Mandatory Gates

| Stage | Name | Required Checks | Minimum Evidence |

| --- | --- | --- | --- |

| S0 | Trigger and Context | PR/MR, merge to main, tag, release request, scheduled scan | actor, commit, branch, ticket, classification |

| S1 | Preflight | format, lint, secret scan, CLAUDE.md rules, file ownership | preflight report |

| S2 | Build | clean build, dependency lock check, deterministic build settings | build logs, artifact digest |

| S3 | Unit and Component Tests | TDD tests, coverage, mutation where applicable, deterministic tests | JUnit/Vitest/pytest reports |

| S4 | Contract and Integration Tests | OpenAPI/AsyncAPI compatibility, Spring Cloud Contract/Pact, Testcontainers | contract test evidence |

| S5 | Architecture Fitness | ArchUnit, dependency graph, banned imports, DDD/context rules | fitness report |

| S6 | Security and Supply Chain | SAST, SCA, secrets, SBOM, Trivy, image scan, license gate | security evidence |

| S7 | AI Artifact Evaluation | prompt tests, guardrail tests, model alias checks, golden dataset eval | AI eval evidence |

| S8 | Package, Sign, Attest | OCI/package build, Cosign signing, SLSA provenance | signature, attestation, SBOM |

| S9 | Deploy and Verify | GitOps promotion, smoke tests, DAST, observability checks | deployment evidence |

| S10 | Evidence Pack Finalization | manifest hash, storage, approval links, retention metadata | evidence manifest |

## 5.1 Blocking Rules

A failed security gate cannot be downgraded by the developer who authored the change.

A missing evidence artifact is a failed gate, even if all tests pass.

A guardrail, OPA, Vault, LiteLLM, Harness, signer, or audit pipeline outage blocks the affected controlled operation.

Any direct model provider SDK import, direct production credential use, or bypass flag is a reject condition.

Auto-Heal may propose remediation through a pull request, but it cannot merge, deploy, or override gates.

# 6. Evidence Pack Architecture

The evidence pack is the standard container for CI/CD proof. It is not a single PDF report; it is a structured collection of machine-readable and human-readable artifacts with hashes, owner, classification, retention, source references, and review status.

Figure 2. Evidence pack flow and authority model.

| Evidence Artifact | Required Contents |

| --- | --- |

| source_context.json | commit, branch, PR/MR, actor, ticket, classification, repository, CODEOWNERS |

| build_report.json | tool versions, dependency lock status, build ID, artifact digest |

| test_results/ | unit/component/contract/e2e/AI test reports and coverage |

| security_results/ | SAST, SCA, secret scan, container scan, DAST, license checks |

| architecture_fitness/ | ArchUnit, dependency graph, banned imports, DDD/port checks |

| sbom/ | CycloneDX or SPDX SBOM for source and container images |

| provenance/ | SLSA attestation, Cosign signature, image digest, registry path |

| deployment/ | GitOps commit, environment, smoke tests, rollback or forward-fix plan |

| approval/ | human approver, CAB/change ticket, risk acceptance, waiver if any |

| evidence_manifest.json | top-level index, hashes, retention, owner, AVCI compliance summary |

# 8. Quality, Security, and Architecture Fitness Gates

## 8.1 Minimum Gate Matrix

| Gate | Requirement | Tools / Evidence | Disposition |

| --- | --- | --- | --- |

| Formatting / lint | Required | Spotless, ESLint, Prettier, Checkstyle, ktlint where applicable | No merge on failure |

| Unit coverage | Required | Changed-line coverage target >= 90%; critical domain logic >= 95% | No merge without waiver |

| Mutation testing | Required for critical logic | PIT, Stryker, or equivalent | Required for finance/security/domain logic |

| Architecture fitness | Required | ArchUnit, dependency graph, Semgrep, custom rules | Reject boundary violations |

| Contract compatibility | Required | OpenAPI/AsyncAPI diff, Pact/Spring Cloud Contract | Reject breaking changes unless versioned |

| SAST/SCA/secrets | Required | SonarQube, Semgrep, OWASP Dependency Check, Gitleaks | Block high/critical |

| Container scan | Required | Trivy/Grype, base image policy | Block high/critical unless approved |

| SBOM/provenance | Required | CycloneDX/Syft, SLSA provenance, Cosign | Required for release |

| OPA/Conftest | Required | Kubernetes, Terraform, Helm, policy checks | Reject policy violations |

| AI evaluation | Conditional | Guardrail tests, model alias checks, golden dataset tests | Required for AI artifacts |

| DAST/smoke | Required before promotion | OWASP ZAP, smoke tests, synthetic transactions | Required for STG/PROD |

## 8.2 Architecture Fitness Function Examples

| Fitness Function | Rule | Enforcement |

| --- | --- | --- |

| No provider SDK | Services must not import direct OpenAI/Anthropic/Gemini/provider SDKs; route through LiteLLM/Spring AI abstraction. | Semgrep import rule |

| No domain-to-infra dependency | Domain package cannot depend on adapter, persistence, HTTP, Kafka, or framework-specific packages. | ArchUnit |

| No cross-context write | A bounded context cannot write to another context schema or table. | Repository review + DB policy |

| Controllers are thin | Controllers delegate to application services; no domain decisions or persistence logic in controllers. | ArchUnit/Semgrep |

| MicroFunction isolation | STP-BUS-* functions must be single-purpose, typed, testable, and framework-envelope compliant. | Unit + ArchUnit |

| Reversibility declared | Mutating flows declare idempotency, compensation, rollback or forward-fix path. | Configuration validator |

| Observability required | Every service emits trace_id, metrics, structured logs, audit intent where required. | OTel test + log scan |

# 9. AI-Assisted Development and Auto-Heal Controls

AI may draft, analyze, propose tests, explain failures, and prepare remediation pull requests. AI must not approve, merge, deploy, override policies, change production data, or bypass guardrails.

Figure 3. Auto-Heal as a guarded pull request, not direct production mutation.

| Control | Requirement |

| --- | --- |

| Prompt disclosure | PR/MR declares AI tool/model used, intent, files affected, and human checker. |

| Classification gate | No secrets, production data, raw PII, or Restricted data in prompts or generated fixtures. |

| Guardrail test | Prompt/guardrail/model changes require Input, Retrieval, Execution, and Output rail tests where applicable. |

| Harness action boundary | AI tool execution beyond sandbox requires Harness, SBAC, OPA, trust score, approval, and audit. |

| Three-attempt remediation cap | AI remediation attempts are capped before human escalation. |

| Learning is reviewed | Auto-Learn creates a candidate; Obsidian/LLM Wiki promotion requires human review and evidence. |

# 10. Environment Promotion and Release Governance

Promotion is not automatic because tests passed. Promotion requires environment-specific evidence, release approval, rollback or forward-fix path, and retained artifacts.

Figure 4. Environment promotion gates.

| Environment | Promotion Condition | Required Evidence |

| --- | --- | --- |

| DEV | Merge to protected branch after required checks | preflight, build, unit, architecture, SAST, evidence skeleton |

| INT | Integration deployment | contract tests, Testcontainers/integration tests, Kafka/schema tests, smoke tests |

| UAT | Business validation candidate | UAT package, known issues, user story traceability, test data classification |

| STG | Production rehearsal | release candidate, DAST, performance smoke, rollback plan, observability readiness |

| PROD | Controlled release | CAB approval, signed artifacts, complete evidence pack, monitoring plan, post-deploy verification |

## 12.2 Retention Rules

Release evidence is retained for the longer of regulatory, audit, contractual, or internal policy requirements.

Failed pipeline evidence is retained for trend analysis and Auto-Learn only after secrets and PII are removed.

Security evidence containing sensitive details must be access-controlled and never copied into public PR comments.

Evidence manifests must contain hashes for every referenced artifact to prevent later substitution.

Superseded evidence remains immutable; corrections are appended through a new evidence record.
