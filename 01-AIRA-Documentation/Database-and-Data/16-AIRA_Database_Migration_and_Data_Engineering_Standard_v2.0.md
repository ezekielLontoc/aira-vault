# 1. Executive Summary

This standard defines how AIRA designs, changes, validates, secures, operates, and improves its data layer. It provides the development baseline for enterprise-grade database work without creating unmaintainable, hardcoded, undocumented, or unsafe data behavior.

Version 2.0 upgrades the v1.0 baseline by incorporating the Enterprise Design Principles and SOLID Enforcement Layer. Data structures, migrations, repositories, pipelines, vectors, configuration tables, audit records, and remediation scripts must preserve bounded-context ownership, Clean Architecture boundaries, idempotency, determinism, fail-safe behavior, reversibility, testability, security posture, observability, and AVCI evidence.

| Strategic Outcome | How v2.0 Delivers It |

| --- | --- |

| Modular data ownership | Each bounded context owns its schema, migrations, RLS policies, repositories, data contracts, and lineage evidence. Cross-context writes are prohibited. |

| Safe change | Database changes use expand/contract migration, compatibility checks, deterministic seed data, forward-fix plans, and rollbackability assessment. |

| AI-native readiness | Embeddings, chunks, prompts, retrieval evidence, model lineage, and re-embedding decisions are governed data artifacts. |

| Maintainability | Naming, audit fields, constraints, transaction boundaries, outbox, idempotency, repositories, and migration patterns are standardized. |

| Compliance and audit | Classification, retention, RLS, encryption, audit evidence, restore drills, and data quality gates are designed into the data layer. |

| Data Is Architecture<br>A table, column, migration, index, vector, seed row, RLS policy, repository method, and data pipeline can change system behavior. Therefore each is an architecture-relevant artifact and must satisfy AVCI and the 20 Enterprise Design Principles. |

# 3. Non-Negotiable Data Principles

| ID | Principle | Operational Meaning |

| --- | --- | --- |

| DB-1 | Schema ownership is explicit | Every table belongs to one bounded context, one owner, and one schema. No orphan schemas or hidden shared ownership. |

| DB-2 | Migrations are the only change path | All DDL, seed data, RLS, extensions, views, functions, and data fixes are delivered through Flyway or approved migration workflow. |

| DB-3 | No shared database ownership | Other services read through APIs, events, projections, or governed read models. They do not write into another context schema. |

| DB-4 | Constraints are business controls | NOT NULL, UNIQUE, FK, CHECK, EXCLUDE, and RLS constraints are required where they represent invariants. |

| DB-5 | Classification is physical | Tables and columns carry classification metadata that governs access, retention, encryption, logs, prompts, and model routing. |

| DB-6 | RLS is defense in depth | Tenant-scoped and restricted data must be protected at the database layer, not only in Java code. |

| DB-7 | Idempotency is stored | All mutation paths have idempotency keys, natural-key constraints, workflow deduplication, or processed-event records. |

| DB-8 | Outbox for cross-boundary effects | Business mutation and event publication intent are committed in one transaction. |

| DB-9 | Forward-only production change | Production rollbacks are forward fixes or compensating migrations unless CAB approves an exception. |

| DB-10 | Redis is not the database | Cache loss must degrade latency, not correctness or auditability. |

| DB-11 | AI data is governed data | Embeddings, chunks, prompts, retrieved evidence, and model outputs are classified artifacts with lineage. |

| DB-12 | AVCI evidence is mandatory | Every migration and data pipeline must prove owner, reason, verification, classification, and improvement path. |

# 5. Enterprise Design Principles and SOLID Enforcement

The 01A Enterprise Design Principles and SOLID Enforcement Layer is a mandatory extension to this data standard. Database artifacts must not bypass application architecture, domain ownership, security, or evidence rules merely because they are implemented in DDL or SQL.

| Principle Area | Database Enforcement Rule |

| --- | --- |

| SOLID | Repositories and data adapters expose narrow interfaces. Domain logic does not depend on PostgreSQL, Flyway, pgvector, or JDBC-specific behavior. Schema objects have single ownership and clear reasons to change. |

| Clean Architecture | Domain and application layers depend on ports/contracts. PostgreSQL, Flyway, and persistence frameworks remain outside the domain core. |

| DDD / Bounded Contexts | Each bounded context owns its schema and migrations. Cross-context reporting uses projections, events, or contracts, not hidden shared writes. |

| Ports and Adapters | Database access is performed through repository ports and persistence adapters. External integrations do not directly mutate domain tables. |

| DRY, KISS, YAGNI | Avoid duplicated lookup tables, over-normalization without purpose, speculative columns, generic EAV tables, and unused migration frameworks. |

| Idempotency by Design | Mutations, event consumers, data imports, migration repairs, and replay actions must be deduplicated and safe to retry. |

| Determinism and Reproducibility | Migrations and seed data produce the same result in clean environments. Time, random, and environment-dependent behavior is controlled. |

| Fail-Safe, Not Fail-Open | If RLS, Vault, OPA, migration validation, audit pipeline, or classification metadata is unavailable, protected data operations are blocked. |

| Human-in-the-Loop | Destructive, high-risk, Restricted, and production-impacting data changes require named human review and approval. |

| Least Privilege / SBAC | DB roles, migration roles, application roles, admin roles, and AI/tool roles are separated and scoped by skill and duty. |

| Observability by Design | Every high-value table, pipeline, migration, and query path produces traceable operational and audit evidence. |

| Reversibility / Rollbackability | Production migration plans define forward-fix, compensation, restore point, validation, and reconciliation approach. |

# 6. Bounded Context Schema Ownership

Schema ownership is the physical expression of AIRA bounded contexts. A service may own one or more schemas, but a schema may not have multiple owning services for write operations.

| Schema Pattern | Standard | Example |

| --- | --- | --- |

| Bounded-context schema | aira_<bounded_context> | aira_document, aira_workflow, aira_config, aira_audit, aira_rag |

| Service write ownership | Only the owning service writes its schema. | document-service writes aira_document; retrieval-service writes aira_rag. |

| Cross-context read | Use API, event, projection, read model, or approved reporting schema. | workflow-service consumes document.created event instead of writing aira_document. |

| Shared reference data | Owned by a named governance service or configuration schema. | aira_reference.country_code; access via versioned contract. |

| Audit schema | Centralized append-only evidence storage. | aira_audit.step_execution_audit, access_decision_audit. |

| Reporting projection | Derivative, rebuildable, and clearly marked non-authoritative. | aira_reporting.loan_status_projection. |

| Rejected Pattern<br>A service must not join to another service owner schema and then update its rows as a shortcut. That pattern violates DDD ownership, ports/adapters, least privilege, and AVCI traceability. |

# 7. Database Design and Naming Standards

| Item | Standard |

| --- | --- |

| Table naming | Singular business noun in snake_case, e.g., document, document_version, process_definition. |

| Primary key | <entity>_id UUID primary key. UUIDv7 preferred where supported; gen_random_uuid acceptable baseline. |

| Foreign key | <referenced_entity>_id with explicit FK constraint unless event-sourced or projection-only. |

| Natural keys | Unique constraints for business identifiers, e.g., tenant_id + reference_id. |

| Audit fields | created_at, created_by, updated_at, updated_by, row_version, trace_id, classification. |

| Timestamps | TIMESTAMPTZ only. Store UTC. Do not use TIMESTAMP WITHOUT TIME ZONE for business/audit time. |

| Amounts | NUMERIC(19,4) or domain-specific NUMERIC precision; never FLOAT/DOUBLE for money. |

| Status fields | Controlled VARCHAR with CHECK or lookup table. Do not use undocumented free-text status. |

| JSONB | Allowed for flexible metadata, evidence envelopes, and source payload snapshots; required keys must be validated. |

| Vector columns | Use vector(n) with explicit embedding_model, embedding_dimension, and embedding_version metadata. |

| Index naming | idx_<table>_<purpose>; uq_<table>_<columns>; fkidx_<table>_<column>. |

| Table Class | Required Columns | Notes |

| --- | --- | --- |

| Business entity | id, tenant_id, natural key, status, classification, created_at/by, updated_at/by, row_version, trace_id | Use optimistic locking and unique natural-key constraints. |

| Reference data | code, name, effective_from, effective_to, status, version, classification, owner | Use temporal validity instead of overwriting historical meaning. |

| Configuration | config_id, config_code, version, status, signed_hash, effective_from/to, owner, classification | Runtime configuration is immutable after activation. |

| Audit/evidence | audit_id, event_time, actor_id/hash, action, resource, decision, trace_id, evidence_ref, classification | Append-only and protected from business-service updates. |

| Outbox | event_id, aggregate_type, aggregate_id, event_type, payload, schema_version, status, retry_count, trace_id | Committed in same transaction as business mutation. |

| Embedding/chunk | chunk_id, source_id, version, text_hash, vector, model, dimension, ACL/classification, lineage | Derivative; rebuildable from Tier-0 source. |

# 8. Flyway Migration Governance

Flyway is the mandatory migration mechanism. AIRA uses migration-as-code to preserve reproducibility, auditability, and deterministic rebuild from a clean baseline.

| Migration Type | Naming / Rule | Required Evidence |

| --- | --- | --- |

| Versioned DDL | VYYYYMMDDHHMM__short_description.sql | Ticket/ADR, owner, purpose, before/after schema diff, tests, rollback/forward-fix plan. |

| Seed/reference data | Versioned migration or repeatable only for derived views/functions | Data owner approval, effective dates, classification, deterministic result proof. |

| Repeatable objects | R__view_or_function_name.sql | Dependency list, idempotent CREATE OR REPLACE pattern, unit/component test. |

| RLS policy | Versioned migration | Threat model, tenant test, bypass-negative test, policy owner approval. |

| Extension change | Versioned migration with ADR | Compatibility check, restore test, upgrade path, fallback/forward-fix plan. |

| Data repair | Versioned migration or controlled operational script | Before/after count, affected keys, approval, audit evidence, reconciliation result. |

| Gate | Control |

| --- | --- |

# 9. Transactions, Idempotency, Concurrency, and Outbox

AIRA treats data mutations as retryable and failure-aware by design. Database patterns must support Temporal replay, Kafka redelivery, API retry, outbox publishing, and MicroFunction compensation without duplicate business effects.

| Pattern | Mandatory Rule | Typical Implementation |

| --- | --- | --- |

| Transaction boundary | One business invariant update must commit atomically. Cross-boundary work uses saga/outbox, not distributed transactions. | @Transactional around domain mutation and outbox insert. |

| Idempotency registry | All external mutation requests and replayable commands require idempotency registration. | tenant_id + idempotency_key + request_hash + response_hash. |

| Optimistic locking | Business entities that can be concurrently updated use row_version or equivalent. | row_version integer incremented per update. |

| Pessimistic lock | Use sparingly for high-contention or strict serialization needs. | SELECT ... FOR UPDATE with timeout and trace evidence. |

| Processed event registry | Event consumers must record consumed event_id/schema_version. | UNIQUE(event_id, consumer_name). |

| Transactional outbox | Domain mutation and event intent commit together. Publisher sends asynchronously. | outbox table + Kafka publisher + retry/DLQ. |

| Compensation | Non-reversible effects require compensating record or workflow. | Temporal compensation activity and audit evidence. |

# 10. Security, RLS, Classification, Privacy, and Retention

| Control Area | Requirement |

| --- | --- |

| Database roles | Separate migration role, application role, read-only role, support role, audit role, and break-glass role. No shared superuser use. |

| Secrets | Database credentials are issued by Vault or approved secrets manager. Static credentials require waiver and rotation evidence. |

| RLS | Tenant and Restricted data must enforce row-level security or equivalent defense-in-depth control. |

| Classification metadata | Tables/columns storing sensitive data must be classified and mapped to retention, logging, masking, and model-routing rules. |

| Encryption | Use approved transport encryption and storage encryption. Highly sensitive values require column-level or application-level protection where required. |

| PII in logs | Queries, errors, migration logs, test reports, and traces must not expose PII, secrets, prompts, or raw document content. |

| Retention | Archive, legal hold, purge, supersede, and retention marker actions must be versioned, audited, and reversible where feasible. |

| Break-glass | Write-capable production data access requires dual approval, time-boxing, session recording/logging, and post-action reconciliation. |

# 11. pgvector, Retrieval Metadata, and AI Data Governance

Vector data and retrieval metadata are governed data, not uncontrolled AI scratch space. Every embedding and chunk must be traceable to Tier-0 source, classification, ACL, extraction version, embedding model, and re-index lineage.

| Artifact | Required Metadata | Governance Rule |

| --- | --- | --- |

| Document chunk | source_system, source_id, source_version, chunk_id, text_hash, extraction_version, classification, ACL, created_at | Chunk is derivative. Rebuild from Tier-0 source when extraction or source changes. |

| Embedding vector | embedding_model, dimension, embedding_version, vector_hash, generated_at, route, owner | Re-embed when model, chunk, classification, or policy changes. |

| Retrieval evidence | query_id, retrieved_chunk_ids, score, rerank_score, policy_result, user/agent, trace_id | AI answer must cite evidence and store retrieval pack reference where required. |

| ANN index | index_type, parameters, build_source, build_time, validation_result | Index is derivative acceleration only. Loss of index must degrade latency, not correctness. |

| Generated summary | source_refs, model_alias, prompt_version, guardrail_result, human_review_status | Not authoritative until reviewed and promoted under knowledge governance. |

| Model-Routing Constraint<br>Confidential or Restricted retrieval content must not be embedded, summarized, or prompted through unapproved cloud routes. Classification-aware routing and redaction must occur before model invocation. |

# 12. Data Engineering and Pipeline Standards

| Pipeline Concern | AIRA Standard |

| --- | --- |

| Source authority | Each pipeline declares Tier-0 source, owner, classification, freshness SLA, and rebuild procedure. |

| Determinism | Transformations must be replayable with the same input, code version, parameters, and reference data. |

| Lineage | Each derived record links to source IDs, source versions, transformation version, execution ID, and trace_id. |

| Schema evolution | Pipelines use explicit schema versions and compatibility tests. Silent column interpretation changes are prohibited. |

| Quality checks | Null, range, referential, duplicate, freshness, reconciliation, and business-rule checks are mandatory for governed datasets. |

| Failure behavior | Bad records are quarantined or DLQ-ed with evidence; pipelines must not silently drop or mutate records. |

| Access control | Pipeline identities use least privilege and SBAC. AI agents do not receive direct database credentials. |

| Observability | Pipeline runs emit metrics, logs, traces, data quality score, reconciliation result, and evidence reference. |

# 13. Testing, Quality Gates, and Architecture Fitness Functions

Database work is tested as engineering work. A migration that compiles but violates ownership, compatibility, RLS, idempotency, or rollbackability is not production-ready.

| Gate | Required Tests / Evidence |

| --- | --- |

| Clean migration test | Apply all Flyway migrations to an empty database and confirm deterministic success. |

| Upgrade migration test | Apply migration to representative previous-version schema and data. |

| Expand/contract compatibility | Current and next application versions work during rolling deployment and deprecation window. |

| Repository component tests | Repositories enforce domain behavior, constraints, errors, and transaction semantics. |

| RLS and tenant isolation tests | Positive and negative tests prove tenant and classification isolation. |

| Idempotency and replay tests | Duplicate commands, events, or retries do not duplicate business effects. |

| Outbox/DLQ tests | Mutation + outbox commit together; publisher retries and DLQ behavior are verified. |

| Performance regression tests | Critical indexes, query plans, and pg_stat_statements thresholds are checked. |

| Vector retrieval tests | Chunk lineage, ACL filtering, similarity threshold, re-index, and stale index behavior are verified. |

| Restore and recovery tests | Backup restore, point-in-time recovery, and Patroni failover evidence are retained. |

| Fitness Function | Rejects |

| --- | --- |

| No cross-context writes | Migration or repository writes to a schema not owned by the service. |

| No raw SQL assembly | String-concatenated SQL using user/config input. |

| No unclassified sensitive columns | Table/column lacks classification metadata for PII, Restricted, or AI-sensitive fields. |

| Mandatory audit fields | Business/configuration tables missing required audit and trace columns. |

| Mutation requires idempotency | POST/PATCH/event consumer path lacks idempotency or deduplication mechanism. |

| RLS required for tenant data | Tenant-scoped table lacks RLS or approved equivalent control. |

| Migration evidence complete | Migration lacks owner, ticket, verification, classification, and forward-fix/compensation plan. |

| Cache not authoritative | Redis/Caffeine data used as sole source of truth or audit evidence. |

# 15. Auto-Heal, Auto-Learn, and Auto-Improve Data Governance

Auto-Heal, Auto-Learn, and Auto-Improve may analyze database signals and propose remediation, but they must not autonomously mutate production data, weaken schema boundaries, bypass RLS, or execute destructive migrations.

| Capability | Allowed Behavior | Required Gate |

| --- | --- | --- |

| Auto-Heal | Detect migration failure, replication lag, outbox backlog, stale index, blocked lock, failed pipeline, or data quality anomaly; draft remediation or PR. | Human approval for production change; tests; rollback/forward-fix plan; evidence; no direct DB credentials for agents. |

| Auto-Learn | Convert resolved defects, migration incidents, query issues, and data quality lessons into reviewed patterns, runbooks, test cases, and wiki entries. | Human review, classification check, source citation, conflict check, and regression evidence. |

| Auto-Improve | Propose index optimization, schema refactoring, data model cleanup, migration template improvement, RLS hardening, or retrieval quality enhancement. | ADR/TDL if material; performance proof; compatibility proof; security review; CAB/Architecture approval as applicable. |

| Auto-Change Constraint<br>No AI-generated data change is accepted unless it preserves or improves SOLID compliance, bounded-context ownership, testability, security posture, observability, rollbackability, and AVCI evidence. |
