# 1. Executive Summary

This document upgrades the AIRA CLAUDE.md Rollout Starter Pack from v2.0 to v3.0. It operationalizes the AIRA CLAUDE.md Reference v3.0 by turning repository rules into executable controls: pre-commit hooks, CI gates, CODEOWNERS, branch protection, compiled lock files, AI tool adapters, onboarding acknowledgements, evidence templates, and audit-ready compliance records.

The v3.0 release is driven by ADR-AIRA-2026-001 and 01A-AIRA Enterprise Design Principles and SOLID Enforcement Layer v1.0. The rollout no longer treats CLAUDE.md as only guidance for AI assistants. CLAUDE.md becomes the human-readable source for machine-enforced repository behavior across human developers, ChatGPT, Codex, Claude Code, IDE assistants, CI pipelines, review workflows, and controlled Auto-Heal / Auto-Learn / Auto-Improve loops.

| Strategic Outcome<br>Every AIRA repository must be self-governing. A developer or AI assistant should know the applicable architecture, security, testing, and evidence rules before editing files. Deterministic violations should fail locally before commit, fail again in CI if bypassed, require the correct human reviewers, and be retained as evidence for audit. |

# 4. Rollout Architecture and Control Flow

The rollout architecture separates the canonical rule source, generated machine controls, local enforcement, CI enforcement, human review, and evidence retention. This separation allows human-readable governance to remain clear while machine checks become repeatable.

| Layer | Component | Responsibility |

| --- | --- | --- |

| Canonical Rule Layer | Root and directory CLAUDE.md templates in registry | Human-readable repository behavior, AI instruction rules, architecture constraints, testing rules, and evidence requirements. |

| Adapter Layer | AGENTS.md, IDE rules, Codex instructions | Tool-specific projections generated from CLAUDE.md without weakening controls. |

| Compiled Control Layer | .claude-md.lock, policy bundle, rule manifest | Machine-readable rule IDs, hash, severity, owner, scope, and enforcement mapping. |

| Local Enforcement | pre-commit hooks and developer workstation tools | Fast deterministic checks before commit: secrets, forbidden imports, direct provider calls, missing templates, rule drift. |

| CI Enforcement | GitHub/GitLab pipeline in strict mode | Full validation: tests, architecture fitness functions, policy checks, SAST/SCA, SBOM, evidence pack, PR/MR compliance. |

| Human Review | CODEOWNERS, branch protection, Maker-Checker | Correct human accountability for architecture, security, AI, database, and domain-sensitive changes. |

| Evidence Layer | OpenKM / Git / CI artifacts / audit store | Retained proof of attribution, verification, classification, improvement path, and principle preservation. |

# 5. Part A - CLAUDE.md Compliance Hook Specification

The compliance hook validates the applicable CLAUDE.md rule stack against changed files and repository-wide controls. It must run locally as a pre-commit/pre-push control and in CI strict mode. Local failure prevents commit. CI failure prevents merge.

| Hook Requirement | Implementation Rule |

| --- | --- |

| Rule discovery | Resolve root CLAUDE.md plus all parent and nearest directory CLAUDE.md files for each changed path. |

| Rule precedence | More restrictive rule prevails. Child rules may add controls but cannot lower severity or relax parent rules. |

| Lock verification | Validate .claude-md.lock hash, source version, template version, rule count, and generated adapter hashes. |

| Diff-scoped checks | Inspect changed hunks for direct model provider imports, bypass flags, raw PII logging, direct database edits, secrets, and missing tests. |

| Repository-scoped checks | Validate required templates, CODEOWNERS, branch protection metadata, policy bundles, and CI workflows. |

| Architecture checks | Trigger or require architecture tests for domain, application, adapter, workflow, security, AI, and data-boundary changes. |

| Evidence check | Reject merge when PR/MR lacks AVCI summary, design-principle impact statement, test evidence, and AI-tool usage declaration. |

| Waiver check | Allow temporary exceptions only with waiver ID, owner, approver, expiry date, risk acceptance, and remediation plan. |

| Rule ID | Example Detection | Default Severity |

| --- | --- | --- |

| AIRA-HOOK-001 | Missing or stale .claude-md.lock | Blocker |

| AIRA-HOOK-002 | OpenAI/Anthropic/Gemini/provider SDK imported directly in application code | Blocker |

| AIRA-HOOK-003 | guardrail_bypass, skip_guardrail, disable_rail, or equivalent flag | Blocker |

| AIRA-HOOK-004 | Secret/token pattern detected in code, docs, prompt, fixture, or test | Blocker |

| AIRA-HOOK-005 | Domain layer imports infrastructure adapter or framework-specific dependency | Blocker |

| AIRA-HOOK-006 | Database migration without forward-fix/rollback note and test evidence | High |

| AIRA-HOOK-007 | Mutation or external-call path without idempotency or retry/compensation evidence | High |

| AIRA-HOOK-008 | AI-generated code declared but missing human checker | High |

| AIRA-HOOK-009 | Business logic added outside STP-BUS-* or approved domain service | High |

| AIRA-HOOK-010 | PR/MR missing design principle impact statement | High |

# 9. Part E - ChatGPT, Codex, Claude Code, and AI Tool Governance

AIRA may use multiple approved AI tools, but there must be one governance source. CLAUDE.md remains the canonical repository rule source. Tool-specific adapters are allowed only as derived projections.

| Tool / Role | Approved Use | Restriction |

| --- | --- | --- |

| ChatGPT Pro - Solutions Architect / IT Head | Architecture synthesis, standards drafting, cross-document reconciliation, senior review, controlled prototype assistance. | No direct production access; no unapproved Confidential/Restricted data in cloud prompts; outputs enter through MR and AVCI gates. |

| ChatGPT Business - Delivery roles | Coding assistance, tests, documentation, PR explanation, troubleshooting, controlled Codex access. | No shared accounts; no secrets/PII/production data; no direct merge or policy override. |

| Codex | Draft code, tests, refactoring suggestions, local/dev checks, PR creation under human supervision. | Cannot approve, merge, deploy, override policy, or perform production-impacting actions. |

| Claude Code | Repository-aware coding assistance using CLAUDE.md hierarchy. | Must obey root and nearest directory rules; no bypass of Harness/OPA/guardrails. |

| IDE assistants / other tools | Limited drafting and explanation when approved by the Technology Stack and security review. | Must consume generated adapter rules; cannot maintain separate weaker instructions. |

| Task agents | Analyze, propose, draft, test, and escalate through Harness/SBAC/OPA. | No direct credentials, no autonomous production change, no self-modifying rules. |

| AI Accountability Rule<br>The named human engineer is the accountable author. The AI tool is a contributor of drafts, suggestions, explanations, tests, or analysis. Reviewer responsibility cannot be delegated to the model. |

# 10. Part F - Onboarding Acknowledgement and Renewal

Before contributing to AIRA repositories, every engineer and approved AI-tool user must acknowledge the repository rule hierarchy, data handling constraints, AI accountability rules, Enterprise Design Principles, and AVCI evidence obligations.

| Acknowledgement Area | Engineer Must Confirm |

| --- | --- |

| Repository hierarchy | I understand root CLAUDE.md, directory CLAUDE.md, generated adapters, and .claude-md.lock precedence. |

| Enterprise principles | I will preserve or improve SOLID, Clean Architecture, DDD, ports/adapters, testability, security, observability, rollbackability, and AVCI evidence. |

| AI usage | I will not use personal or unapproved AI accounts for AIRA code, data, prompts, architecture materials, or documents. |

| Data handling | I will not paste secrets, raw PII, production data, Restricted content, credentials, or tokens into AI tools, tests, logs, examples, or docs. |

| Human accountability | I remain accountable for AI-assisted outputs that I commit, submit, or approve. |

| Testing and evidence | I will provide required tests, security evidence, classification evidence, and AVCI summary before merge. |

| Escalation | I will escalate ambiguity, conflicts, high-risk changes, or rule violations rather than bypass controls. |

Renewal is required quarterly, on material repository rule change, after significant incident, after privilege change, and before a user receives elevated AI, repository, security, database, or deployment authority.

# 11. Enterprise Design Principles Enforcement Matrix

| Principle | Rollout Enforcement |

| --- | --- |

| SOLID | ArchUnit/Sonar rules; class responsibility review; dependency inversion checks. |

| Progressive Autonomy | Trust score, risk tier, bounded action vocabulary, approval escalation. |
